# Specification: Spotify-to-Soulseek Watchlist Automation

**Track ID:** spotify-watchlist_20260305
**Type:** Feature
**Created:** 2026-03-05
**Status:** Draft

## Summary

Implement a "set-it and forget-it" automation system that monitors Spotify playlists and "Liked Songs" for new additions, automatically enqueuing them for high-quality acquisition via Soulseek (slskd) based on user-defined quality profiles.

## Context

Djinn NETRUNNER provides a complete system for automated music discovery, download, organization, and streaming. This track will finalize the automation loop, moving from manual "Sources" to an automated "Watchlist" system, ensuring high-fidelity acquisition without manual intervention.

## User Story

As an avid music collector, I want to be able to sync my Spotify playlists and liked songs for download so that I can have a "set-it and forget it" workflow for growing my local library.

## Acceptance Criteria

- [ ] **Watchlist Management:** A new UI section allows users to register Spotify playlist URLs or enable "Liked Songs" tracking.
- [ ] **Quality Profiles:** Users can select acquisition profiles (e.g., "Lossless Only", "Preferred FLAC", "320kbps MP3") per watchlist.
- [ ] **Automated Polling:** A background worker (Go) periodically checks watchlists for new tracks added since the last snapshot.
- [ ] **Intelligent Deduplication:** The system checks the local filesystem and active `slskd` queue before creating new acquisition jobs.
- [ ] **Sync Dashboard:** A real-time view showing the last sync time, tracks found, and acquisition status for all watched sources.

## Dependencies

- **Existing Code:** `ops/worker/job_handlers.py` (Spotify parsing logic) and `ops/worker/slskd_client.py`.
- **Infrastructure:** Requires `SPOTIFY_CLIENT_ID` and `SPOTIFY_CLIENT_SECRET`.

## Out of Scope

- Browsing/Managing private Spotify profiles of other users.
- Mobile-specific UI commands or native mobile app components.
- Automatic metadata correction/tagging (handled by existing metadata service).

## Technical Notes

- Implement the polling logic in the Go backend using a ticker-based service.
- Use the new SQLite abstraction for storing "last_synced_at" and "snapshot_id" for each watchlist.
- Leverage the existing `SlskdService` in Go for enqueuing searches.
