# Specification: Architecture Evolution & Optimization

**Track ID:** architecture-evolution_20260304
**Type:** Refactor/Feature
**Created:** 2026-03-04
**Status:** Draft

## Summary

Evolve NetRunner's architecture to be a standalone, high-performance application comparable to Lidarr. This involves migrating from PostgreSQL to SQLite for portability, implementing a robust metadata cache to remove API dependencies, optimizing IO/concurrency, and adding privacy controls for P2P acquisition.

## Context

Current research identifies NetRunner as "fragile" due to its heavy reliance on external services (PostgreSQL, MusicBrainz) and inefficient concurrency patterns. To become a viable, deployable product, it needs to be self-contained and resilient.

## Acceptance Criteria

- [ ] **Standalone Persistence:** Migrate from PostgreSQL to SQLite (with WAL mode) using GORM. Application must run with zero external DB dependencies.
- [ ] **Metadata Resilience:** Implement a local "Shadow Cache" for MusicBrainz/Spotify data to prevent rate-limit stalls.
- [ ] **Optimized Concurrency:** Replace Postgres advisory locks with an in-memory or SQLite-based job queue (e.g., Asynq with embedded Redis or a custom SQLite-based queue optimized for high-throughput).
- [ ] **IO Performance:** Parallelize metadata extraction and file hashing using worker pools.
- [ ] **Privacy:** Implement SOCKS5/HTTP proxy support for the `slskd` client.

## Dependencies

- **Complete Track:** `web-api-migration_20260304` (Web API must be in Go before switching DBs)
- **Libraries:** `modernc.org/sqlite` (pure Go), `github.com/dhowden/tag`.

## Out of Scope

- Cloud sync features.
- Multi-node clustering (standalone focus).

## Technical Notes

- **SQLite Migration:** Use GORM's SQLite driver. Enable WAL mode for concurrency.
- **Metadata Cache:** Store JSON blobs of API responses with a TTL.
- **Queue:** Since we are removing external deps, "Asynq" (Redis) is now a liability. We must implement a robust, persistent SQLite-backed priority queue.
