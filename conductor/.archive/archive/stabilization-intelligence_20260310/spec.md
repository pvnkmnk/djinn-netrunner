# Specification: Stabilization & Intelligence

**Track ID:** stabilization-intelligence_20260310
**Type:** Feature/Refactor
**Created:** 2026-03-10
**Status:** Draft

## Summary

Enhance the NetRunner acquisition pipeline with advanced resilience, deep media server integration, and a production-grade data layer supporting multi-node coordination.

## Context

NetRunner has successfully migrated to Go and SQLite, but currently operates as a single-appliance system with basic retry logic and secondary cover art handling. To meet the original vision of a robust "streaming appliance," the system needs to intelligently handle transient failures and ensure library integrity with high-fidelity metadata.

## Acceptance Criteria

- [ ] **Advanced Backoff:** Job items for search/acquisition implement exponential backoff (e.g., 1m, 5m, 1h, 1d) with maximum attempt limits.
- [ ] **Smart Refresh:** Before attempting a track acquisition, the system queries the Gonic API to verify if the track is already "Online" and indexed.
- [ ] **Cover Art Embedding:** Downloaded cover art is embedded directly into the ID3/FLAC tags of the acquired file during the import phase.
- [ ] **Production Data Layer:** A finalized database architecture (e.g., LiteFS or similar distributed SQLite pattern) is implemented to support multi-node coordination.

## Dependencies

- **Gonic API:** Access to Subsonic REST endpoints (`getSong`, `search3`).
- **Tagging Library:** Extension of `dhowden/tag` or integration of a writer-capable library like `go-taglib` or `id3-go`.
- **Database Coordination:** Research into LiteFS, rqlite, or custom locking mechanisms.

## Out of Scope

- Full redesign of the UI (handled in `frontend-polish`).
- Support for media servers other than Gonic (Subsonic-standard only).

## Technical Notes

The database refactor is the highest risk. We will utilize a phased "Research -> Prototype -> Migrate" approach for the multi-node coordination to ensure no data loss and maintain standalone portability.
